## 2. 核心 Prompt 迭代记录

在处理任务 4时，单纯依赖 AI 导致逻辑偏差，以下是prompt的迭代过程：

* **初代 Prompt：** > “请帮我计算任务4的高峰小时系数。先找出刷卡量最大的一小时，然后在这个小时内，分别以5分钟和15分钟为窗口找最大值，套用公式计算 PHF。”
* **AI 生成的风险代码与问题：** AI 使用了 `.rolling('5min').count().max()`来寻找最大值。我意识到这与要求相悖**，而 `rolling` 会产生重叠的滑动计算，这会导致业务定义的根本性错误。
* **优化后的高阶 Prompt：** > “在任务4中，你的 `rolling` 逻辑不符合交通工程的固定时间桶定义。请你修改算法架构：
    > 1. 先用 `idxmax()` 锁定确切的高峰小时。
    > 2. 将该小时的数据切片强制转为 `DatetimeIndex`。
    > 3. **核心修正：** 必须使用 `.resample('5min').size()` 和 `.resample('15min').size()` 进行离散的降采样聚合，确保时间桶互不交叉。
    > 4. 在最终的公式推导处，用中文详细注释为何除数是 12和 4。”

## 3. Debug 记录

* **报错现象：** 执行预处理阶段时，Pandas 抛出致命错误 `KeyError: '交易时间'`，且 `df.info()` 显示列数为 1，整体被解析成了超级长的一列。
* **深度归因与解决：** * **排查：** 作业文档宣称数据集为制表符（`\t`）分隔，但我通过记事本直接读取 `ICData.csv` 裸数据，发现数据实则被逗号（`,`）分隔。这导致 Pandas 找不到 `\t` 而合并了表头。
    * **次生报错：** 修正分隔符后，遭遇 `ValueError: time data does not match format`。进一步比对发现，真实数据的时间戳尾部携带了“秒数”（`11:45:12`），而需求文档只给了 `HH:MM`。
    * **最终修复：** 我手动修改了代码入口，将 `sep='\t'` 覆写为 `sep=','`，并将时间格式化解析掩码严格修正为 `format='%Y/%m/%d %H:%M:%S'`。
